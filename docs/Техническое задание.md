# ТЗ: Учебная платформа с поэтапным открытием уроков (Laravel, MVP)

**Проект:** учебный проект для ученика  
**Цель:** реализовать механику “следующий урок открывается после прохождения предыдущего”  
**Стек:** Laravel + Blade + MySQL/PostgreSQL  
**Дата:** 2026-04-23

---

## 1) Цель и результат

Реализовать учебную платформу, где пользователь проходит уроки по порядку.  
После завершения текущего урока открывается следующий.  
Пользователь видит свой прогресс по курсу.

---

## 2) Что должно работать (MVP)

1. Регистрация и вход пользователя.
2. Список курсов.
3. Страница курса с модулями/уроками.
4. Статусы уроков:
   - `completed` — пройден,
   - `available` — доступен,
   - `locked` — заблокирован.
5. Поэтапный доступ:
   - первый урок доступен всегда;
   - урок `N` доступен только если завершен `N-1`.
6. Кнопка “Завершить урок”.
7. Расчет и показ прогресса курса в процентах.

---

## 3) Что не входит в MVP

- Квизы/тесты и автопроверка.
- Платежи и подписки.
- Сертификаты.
- Сложная админка.
- Видео-хостинг/стриминг.

---

## 4) Пользовательские роли

### Student
- Просмотр доступных уроков.
- Прохождение уроков.
- Просмотр собственного прогресса.

### Admin (минимально)
- Создание/редактирование курсов, модулей и уроков (через seed/БД на старте).

---

## 5) Бизнес-правила доступа

### Правило открытия (для MVP)
- Все уроки в курсе имеют порядок `position`.
- Доступ к уроку:
  - если это первый урок (`position = минимальный`) — доступен;
  - иначе доступен только при завершении предыдущего урока.

### Поведение при попытке открыть заблокированный урок
- Сервер возвращает редирект на страницу курса.
- Показывается сообщение: “Сначала завершите предыдущий урок”.

---

## 6) Маршруты (пример)

- `GET /courses` — список курсов  
- `GET /courses/{course:slug}` — страница курса  
- `GET /courses/{course:slug}/lessons/{lesson:slug}` — просмотр урока  
- `POST /lessons/{lesson}/complete` — отметить урок как завершенный

## 7) Структура БД

## Таблица `courses`
- `id`
- `title` (string)
- `slug` (string, unique)
- `description` (text, nullable)
- `is_published` (boolean, default true)
- `created_at`, `updated_at`

## Таблица `modules`
- `id`
- `course_id` (FK -> courses.id, cascade delete)
- `title` (string)
- `position` (int)
- `created_at`, `updated_at`

## Таблица `lessons`
- `id`
- `course_id` (FK -> courses.id, cascade delete)
- `module_id` (FK -> modules.id, nullable, cascade delete)
- `title` (string)
- `slug` (string)
- `content` (longText/text)
- `position` (int, порядок в рамках курса)
- `is_published` (boolean, default true)
- `created_at`, `updated_at`

**Ограничения/индексы:**
- `unique(course_id, position)`
- `unique(course_id, slug)`

## Таблица `user_lesson_progress`
- `id`
- `user_id` (FK -> users.id, cascade delete)
- `lesson_id` (FK -> lessons.id, cascade delete)
- `status` (string/enum, MVP: `completed`)
- `completed_at` (datetime, nullable)
- `created_at`, `updated_at`

**Ограничения/индексы:**
- `unique(user_id, lesson_id)`
- `index(user_id, status)`

## 8) Eloquent-связи

- `Course` hasMany `Module`
- `Course` hasMany `Lesson`
- `Module` belongsTo `Course`
- `Module` hasMany `Lesson`
- `Lesson` belongsTo `Course`
- `Lesson` belongsTo `Module` (nullable)
- `User` hasMany `UserLessonProgress`
- `UserLessonProgress` belongsTo `User`
- `UserLessonProgress` belongsTo `Lesson`

## 9) Backend-логика (обязательно на сервере)

## Проверка доступа к уроку (псевдокод)

```php
function canAccessLesson(User $user, Lesson $lesson): bool
{
    $previous = Lesson::where('course_id', $lesson->course_id)
        ->where('position', '<', $lesson->position)
        ->orderByDesc('position')
        ->first();

    if (!$previous) {
        return true; // первый урок
    }

    return UserLessonProgress::where('user_id', $user->id)
        ->where('lesson_id', $previous->id)
        ->where('status', 'completed')
        ->exists();
}
```

## Завершение урока

- По кнопке “Завершить урок” отправляется POST-запрос.
- Используется `updateOrCreate` по `(user_id, lesson_id)`:
  - `status = completed`
  - `completed_at = now()`
- Повторное завершение не должно создавать дубликаты.

## Расчет прогресса курса

```text
progress_percent = floor(completed_lessons / total_lessons * 100)
```

## 10) UI/UX требования

## Страница курса
Для каждого урока отображать:
- название;
- статус (completed/available/locked);
- кнопку “Открыть” только для `available/completed`;
- для `locked` — неактивный элемент.

## Страница урока
- Заголовок и контент урока.
- Кнопка “Завершить урок” (если еще не завершен).
- Если завершен: “Урок пройден”.
- (Опционально) Ссылка “Следующий урок”.

## Сообщения
- Успех: “Урок отмечен как пройденный”.
- Ошибка доступа: “Этот урок пока заблокирован”.

---

## 11) Безопасность и консистентность

1. Только авторизованный пользователь может проходить уроки.
2. Нельзя завершить урок от имени другого пользователя.
3. Нельзя обходить блокировку через прямую ссылку.
4. Проверка доступа выполняется на backend (middleware/policy/service).
5. Уникальность прогресса обеспечивается БД + `updateOrCreate`.
6. Студентам показываются только опубликованные уроки/курсы (`is_published = true`).

## 13) Acceptance Criteria (критерии приемки)

1. Пользователь после входа видит курс и список уроков.
2. Первый урок доступен сразу.
3. Второй урок заблокирован до прохождения первого.
4. После завершения первого урока второй становится доступным.
5. Прямой переход к заблокированному уроку запрещен (редирект + сообщение).
6. Прогресс считается корректно.
7. Повторная отметка завершения не создает дубль в БД.

---

## 14) Обязательные тест-кейсы

1. **Новый пользователь**  
   Ожидание: доступен только первый урок.

2. **Попытка открыть урок №3 сразу**  
   Ожидание: отказ в доступе, редирект на курс.

3. **Завершение урока №1**  
   Ожидание: урок №2 становится доступным.

4. **Повторное завершение урока №1**  
   Ожидание: дубликатов в `user_lesson_progress` нет.

5. **Неавторизованный доступ к уроку**  
   Ожидание: редирект на страницу логина.

---

## 15) Риски и меры

1. **Неправильный порядок уроков (`position`)**  
   Решение: уникальный индекс + валидация при создании уроков.

2. **Сложная логика в контроллерах**  
   Решение: сервисы `LessonAccessService`, `CourseProgressService`.

3. **Обход ограничений через фронт**  
   Решение: считать backend единственным источником истины.
