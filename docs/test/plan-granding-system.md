# Система оценки практических заданий в песочнице (10-балльная)

## Цель
Добавить в учебную платформу автоматическую и полуавтоматическую проверку практических заданий с итоговой оценкой **только по 10-балльной шкале**.

---

## Варианты реализации

## Вариант A — Чеклист-оценка (MVP, самый быстрый)
Подходит для старта за 1–2 спринта.

### Как работает
1. Для каждого задания преподаватель задаёт набор критериев (чеков), например:
   - в HTML есть `.card`;
   - у `.card` есть `width: 300px`;
   - у `.card` есть `padding: 20px`;
   - у `.card` есть `border: 2px solid #ccc`;
   - блок центрирован (`margin: 50px auto`).
2. Проверяющий движок исполняет тесты в sandbox.
3. Каждый чек даёт фиксированный вес (например 2 балла из 10).
4. Итог = сумма весов по пройденным критериям (**0–10**).

### Плюсы
- быстрый запуск;
- прозрачно для ученика (видно, что именно не пройдено);
- легко поддерживать.

### Минусы
- ограниченная гибкость для творческих заданий.

---

## Вариант B — Тестовый раннер (лучший баланс)
Подходит для фронтенд-практики и задач по JS/DOM.

### Как работает
1. Для задания хранится набор автотестов (например Playwright/Jest для sandbox-сцены).
2. При нажатии «Проверить» код ученика прогоняется в изолированной среде.
3. Тесты возвращают:
   - pass/fail;
   - вес теста;
   - сообщение для ученика.
4. Балл = сумма веса пройденных тестов, нормализованная к **10 баллам**.

### Примеры проверок
- наличие нужной DOM-структуры;
- точные CSS-правила;
- интерактив (по клику меняется класс/текст);
- отсутствие ошибок JS в консоли.

### Плюсы
- очень точная проверка;
- легко масштабируется на новые уроки;
- близко к реальной инженерной практике (TDD mindset).

### Минусы
- потребуется инфраструктура для исполнения тестов и таймаутов.

---

## Вариант C — Гибрид: автооценка + ревью преподавателя
Подходит для продвинутых/творческих задач.

### Как работает
- 70–80% балла даёт автопроверка;
- 20–30% добавляет преподаватель по рубрике (чистота кода, читаемость, UX).

### Плюсы
- сохраняет объективность;
- учитывает качество, которое тяжело формализовать.

### Минусы
- требует времени преподавателя.

---

## Рекомендуемая архитектура (для вашего проекта)
С учётом уже существующих моделей прогресса и попыток тестов (`lessons`, `user_lesson_progress`, `user_quiz_attempts`) лучше внедрить отдельный контур практик.

## 1) Схема данных

### `lesson_practices`
- `id`
- `lesson_id` (FK)
- `title`
- `description`
- `max_score` = `10`
- `pass_score` (например `7`)
- `is_active`

### `practice_checks`
- `id`
- `lesson_practice_id` (FK)
- `type` (`css_rule`, `dom_selector`, `js_behavior`, `snapshot`, ...)
- `payload` (JSON-описание проверки)
- `weight` (вклад в итог)
- `sort_order`

### `practice_submissions`
- `id`
- `user_id` (FK)
- `lesson_practice_id` (FK)
- `html_code`, `css_code`, `js_code` (или ссылка на blob)
- `raw_result` (JSON ответа раннера)
- `score` (0.0–10.0)
- `passed`
- `attempt_no`
- `checked_at`

### `practice_check_results`
- `id`
- `practice_submission_id` (FK)
- `practice_check_id` (FK)
- `passed`
- `earned_weight`
- `message`

## 2) Поток проверки
1. Ученик нажимает **«Проверить задание»**.
2. Бэкенд создаёт `practice_submissions` со статусом `pending`.
3. Очередь (Laravel Queue) отправляет задачу в sandbox-раннер.
4. Раннер исполняет проверки, возвращает JSON.
5. Бэкенд агрегирует балл в диапазоне **0–10**, сохраняет результат.
6. Если `passed = true`, обновляется прогресс урока.

## 3) Формула оценки (единая)
- `score_10 = round((earned_weight / max_weight) * 10, 1)`

Это делает систему простой и одинаковой для всех уроков.

---

## Анти-абьюз и надёжность
- лимит времени выполнения (например 3–5 сек на проверку);
- лимит памяти/CPU в sandbox;
- троттлинг попыток (например не чаще 1 раза в 10 секунд);
- версионирование тестов (`check_version`), чтобы корректно пересчитывать старые результаты;
- логирование ошибок раннера и трассировка по `submission_id`.

---

## UX-идеи
- показать «Что выполнено / Что не выполнено» по каждому критерию;
- кнопка «Показать подсказку» после N неудачных попыток;
- бейджи: «С первой попытки», «Идеальное решение 10/10»;
- прогресс по практикам в модуле;
- отображение **лучшей** и **последней** попытки.

---

## Пошаговый план внедрения

## Этап 1 (MVP, 1–2 недели)
- таблицы `lesson_practices`, `practice_checks`, `practice_submissions`;
- 3–5 типов базовых проверок (DOM + CSS);
- кнопка «Проверить» в интерфейсе урока;
- вывод балла по шкале **/10** и статуса pass/fail.

## Этап 2
- детальный отчёт по каждому check;
- ограничение частоты попыток;
- журнал попыток ученика.

## Этап 3
- JS-поведенческие тесты;
- шаблоны заданий для преподавателя;
- гибридный режим с ручной дооценкой.

## Что выбрать прямо сейчас
Для вашего кейса оптимально:
1. начать с **Варианта A** (чеклист),
2. архитектуру сразу закладывать как у **Варианта B**,
3. позже включить элементы **Варианта C** для сложных задач.

Это даст быстрый запуск оценок без переписывания платформы и сразу закрепит единый стандарт **10-балльной оценки**.
